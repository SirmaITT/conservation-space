Create New iDoc

A new iDoc is created either by selecting an existing template or from scratch.

To create a new iDoc in a case the user should have the necessary permissions.

  • The user selects the action "Create Document" from:
    • Case Details/ Documents Tab/ "Create Document" action (1-2-3) of a selected document section of a selected case.
    • Project/ Documents Library section/ "Create Document" action in each document folder. (TBD)

  • The system opens a screen with a list of the available templates for the project/ case.. 

In the list with templates there could be a template with a name "Empty Document" - this is a document with only a few default widgets. The Empty document is also a start point for creation of other templates and could be different for the different organizations. 

The user selects a template from the list (1-2), enters a description of the new document (3) and clicks the button "Create" (4).

  • The system opens the screen for document creation.  By default the primary template for the selected document type is loaded when the document is selected for creation.

    •  the user may select another template from the list with templates (1-2) with sections and widgets as the ones predefined in the selected template.


The user could insert new widgets/ sections, delete existing widgets/ sections in the template, change formatting etc. (For details see iDoc Content Management topics).

The screen is divided into 3 parts:

    • Document Tree View (1) - in  the left part of the screen, where each section from the document content is displayed. The user could navigate in the document content by just selecting a section from the tree view.
    • Document Content (2) - in the middle part of the screen, where the user could manage the document content by adding/ removing text sections and widgets, entering a text with the free text editor, inserting links to other objects, etc. (For details see iDoc Content Management). The template selected by the user (here "Empty Document")  is loaded into this part.
    • Comments (3) - in the right part of the screen. Here the user is able to create and view comments on the document as an object and each one of its sections. The section is empty with the document creation.

The empty document has a default section with the following widgets in it:

    • Document identification (or Title) widget (4) – the widget contains the ID, Title, creator and date of creation of the document, version number (new document is created with version 0.1).
    • Text editor (5) -  the default section has a text editor in it
    • Document Properties (6) widget – the section contains selected attributes of the document (Title, Description, Type, Created from, Created on, Last modified from, Last modified on, etc) (7). In the section there is a link "View more ..." and the user could review all attributes of a document. The user is able to edit some of the properties in the section (10) depending on his/her permissions.
    • Version History widget (8) – the section contains information for version number, date of creation, created by, description of change. The document version is displayed as a link, and when the link is selected the history version of the document is opened for review. The section is empty with document creation.
    • Relationships widget  (9) - the section contains information for all objects, related with the document, no matter what is the relationship type (parent/ child, peer-to-peer).

  • The user enters a title of the document from the Title section (1) or from the Properties section. The document could not be saved if the Title field is empty.
    There is no restriction in title uniqueness. Documents with the same title may exist in the system.

The user enters text using the text editor (2) and saves the document (3) or cancels the operation. For details see Save iDoc and Cancel iDoc topics.


  • The new iDOC is created with sections and widgets as the ones predefined in the template. All changes in the document content are also saved in the corresponding sections (1-2).

The system has automatically filled in the data in the sections Details (3), Version History (4), Relationships (5).

When a new iDoc is created and saved for first time, the following properties in the document Properties section are set automatically:

      • Unique ID - the unique document identifier in the system
      • Title - If the user does not change the title form the "Properties" section, the title of the document is set automatically to the name of the template.
      • Created by - the user who created the document
      • Created on - the date and time the document has been created
      • Version - the version of the document is set to 1.0
  • The action is recorded in the Audit Log.

  • The user could insert new widgets/ sections, delete existing widgets/ sections in the template, change formatting etc. (For details see iDoc Content Management topics).
  • The created iDoc is visible in the corresponding document section of the case (1-2-3).

  • The user could create a new iDoc as a sub-document of an existing iDoc.
    • Create document action is active only for documents in Preview mode.
    • The user selects the action from the list of available actions of the iDoc(1-2).

    • A screen with available templates opens, the user enters the necessary data and saves the newly created document.
    • The two documents are in parent-child relationship.
    • When a parent document is published, then it is published with all its children.

An iDoc could have unlimited number of children iDocs (grand-children, etc.)

Rules for Document Identifiers

The system is able to support user defined rules for generation of different unique identifiers for different document types and the identifier to be automatically generated based on these rules. 

The rules for each part of the identifier and how it should be generated are specified and described for each document type in its definition. The rules include:

  • Fixed string – this could be any free sequence of characters – letters, numbers (CR, DRA, FM!@3, etc.)
  • Property that will participate in the identifier – this could be:
    • Current document specific properties - I may want to include the Creation Date of the document in the identifier
    • properties of the parent project - I may want to include in the identifier of the document the ID of the parent project
    • properties of the parent case
    • properties of other document in specific relationship with the current document – this will be valid only for automatically created documents and automatically created relationships between them and other documents. For example document of type document release authorization (DRA) is automatically generated when ISO document is approved. The newly created document is in relationship “is DRA of” with the approved ISO document. I want the ID of the DRA to be DRA_XXX-XXX-XX-XXXX, where XXX-XXX-XX-XXXX is the ID of the approved ISO document.
  • When property is selected from a code list it is possible to define if its code or description is used in the identifier (ex. ENG (code) or Engineering Department (description))
  • Sequence number - it is possible to define how the sequence number is counted. For example when a new document ID is created then count all documents which have the same property A, or count all document that have the same properties A+B, or the same properties A+B+C
  • Separator – it is possible to define the separators between different parts of the identifier. Different separators could be used between the different parts (“_”,” – “,” /”, “.”, “a”)

Example 1: Document of type “e-ISO Document” should have an identifier of type: XXX-XXX-XX-XXXX (ENG-AED-CH-0001, ENG-AED-FM-0001). In the definition is specified:

  1. Department – XXX – mandatory property of a document. (expl ENG)
  2. Functional Group-XXX - mandatory properties of a document. (expl AED)
  3. Document Type-XXX - mandatory properties of a document. (expl CH)
  4. Number of document-XXXX – is the sequence number of the document of this type for this functional group and department (the sequence number is generated for documents that are from the same department, functional group and type – property 1 and 2 and 3)

When the user selects action "Create New Document" of type “e-ISO Document” and enters the mandatory properties, then the system generates a document identifier with ID containing the next sequence number of this document type.( if ENG-AED-CH-0002 exists the newly created document  will be with ID: ENG-AED-CH-0003).

Example 2: Document of type User Story should have an identifier of type: XXX-YYY-ZZZ-USNNN. In the definition is specified:

  1. XXX – is the code of product ID – “Product” is property of the user story
  2. YYY – is the code of the Theme of the user story – “Theme” is property of the user story
  3. ZZZ – is the code of the epic of the user story – “Epic” is property of the user story
  4. US – is string
  5. NNN – is the sequence number of the user story for the selected Epic
  6. “-“ is separator

In the system there are 10 user stories for the selected epic.

When the user selects action "Create New Document" of type “User Story” and enters the following mandatory properties:

  • EMF – product
  • DCM – document management as Theme
  • CRD – create document as Epic

Then: A document is generated with ID: EMF-DCM-CRD-US011

Mandatory Data Properties for Specific Document Types

Some document types may be configured with mandatory data properties. Values for these properties should be entered before the creation of the document.

The mandatory data properties could be selected from existing code lists or to be simple text values.  

Example:

  • The user selects the action "Create document" in the official section of selected case.
  • The system opens the "Document creation" picker (1) and the user should select the document type from the drop down menu (2).

  • The user selects the type of document.
  • The system opens the form in which for the selected type ("Document for testing(2)") there are fields for mandatory properties:
    • Description (1) is not a mandatory field.
    • Test Field 1 (2) is a mandatory field - free text may be entered.
    • Test Field 2 (3) is a mandatory field - free text may be entered.
    • Test Field 3 (4) is a mandatory field - free text may be entered.
    • Created on (5) is a mandatory field. The automatically entered date of creation cannot be edited by the user.
    • Test Codelist (6) is a mandatory field. The user should select a value from the corresponding drop down list.
  • The user has options:
    • not to fill in data in all mandatory fields  and click Skip (7)
    • to enter data in all mandatory fields and click OK (8)
    • to cancel the operation (click the button Cancel)


  • When the user fills in all mandatory data fields and clicks "OK" - the entered data will appear as property of the document and the document may be created.
  • When the user does not fill in all data fields and clicks "Skip" :
    • The system opens the form of the document where the user enters the title of the document (1) and when tries to save the document (2) the system displays error message (3) and if the user does not fill in the mandatory data the document cannot be created.



In another circumstance, for example when creating a case, the information in the mandatory fields may come from a code list server and thus when an option in Test Field 1 is selected, a certain, specific, sub-set of options are available in Test Field 2 and in Test Field 3.  

Create New Document as a Sub-document of the Main Document

A user can create a document as part or as a child of another document. The user must have the necessary rights.

  • The user has two options to create a document from within the main document (or a sub-document): 1) from landing page of the main document.

Or 2) from document's actions in the Documents and Media section of the case or from My/Project/Case dashboard.

  • Next the user must select the document type from the drop-down menu:

  • The user confirms their choice by selecting the OK button.

  • Then the user types the title of the new document (sub-document) and saves it by selecting Save and Close.

The new document (sub-document) is in relation Part of with the main document (the user may need to drag-and-drop the Object data and the Relationships widgets).

The main document is in relation Has child with its sub-document (the user may need to drag-and-drop the Object data and the Relationships widgets).

  • The same steps are applicable for a document which is created by uploading an image or a text document.
