Sirma Platform : Role Manager in a Library and in Library Objects

User with role Manager in a library can not be changed or removed on any of the objects part of the library.

The role of Manager on library level should be changed by the Administrator in order to be changed on the objects in the library.

  • Management Permissions section in the Manage Permissions screen
    • No Management Permissions section on the ontology class (here - Library) landing page
    • Management Permissions section on objects in the library
      • The permissions in the Management Permissions section could not be edited
  • The permissions in the Management Permissions section are with the highest priority
    • When the role Manager on library level is given to another user, then the special and inherited permissions (of the former manager) become valid

Management Permissions section in the Manage Permissions screen

In the section are displayed all users with the roles Manager on library level.

No Management Permissions section on the ontology class (here - Library) landing page

  • The user (1) opens the permissions tab of a objects library (2) (for example Mark).
    • The action Edit Permissions is active (3) if the user has edit permissions on the library (4).
  • There is no Management Permissions section in the Permissions on library level.

Management Permissions section on objects in the library

Any of the objects (1) part of the library have the following permission sections visible when tab Permissions (2) is opened - Special Permissions, Inherited Permissions, Management Permissions.

  • The user opens the permissions tab of an object in the library (1-2).
    • If the user has edit permissions on the project then the action Edit Permissions is active.
    • The Permissions type is Inherited (3) (no Special permissions are given to users/ group for that object).
    • There is Management Permissions section on objects level (4). The user with the role Manager in the project is visible in the section (5).

The permissions in the Management Permissions section could not be edited
  • The user with edit permissions on the object clicks the button "Edit Permissions" and the permission sections are opened in edit mode (1).
  • The Management Permissions section on the object level (2) is not editable.

The permissions in the Management Permissions section are with the highest priority

Example:

If an user is with the role Manager on a library and with Special permissions Consumer on an object from this library, then the special permissions on the object are not taken into account.

  • The user opens the permissions tab of an object in the library.
    • The Permissions type is Inherited and Special (1).
      • In the Special Permissions section the user (user3) is set with Consumer permissions (2).
      • In the Management Permissions section the same user is with Manager permissions on the library (3).

  • The user opens the list of available actions on the object in the library (1-2-3-4-5-6-7). The list displays all actions for the role Manager (not the actions for a consumer).

When the role Manager on library level is given to another user, then the special and inherited permissions (of the former manager) become valid

  • The role of Manager on the library (1) (here - Mark) is given to another user (2).

  • The user who was Manager of the library (1)  and with consumer permissions set in the Special Permissions section opens the object details page (2)
    •  His/ her role in the Special Permissions section is Consumer (3).
    • The Management Permissions for the library are given to another user (4). 
  • The Special permissions of a Consumer, given to the user (user3), former manage of the library, are valid - the list of available actions is for a consumer on the object (5-6).
    • The consumer has no permissions to edit the Permissions on the object or to edit the object.
