Task as a Sub-task of another Task

Preconditions 

  1. Only the task assignee and users with the role of "Manager" on the project could create sub-tasks of a task. 
  2. Sub-task could not be created in a task in status "Completed". 
  • The user selects from Task landing page/Actions/ Create Sub-task (1-2).

  • The user fills in the necessary data (1-2-3) and clicks "Create Sub-task" (4).


  •  The newly created task is  added in the "Sub-tasks" section of the task from where the operation was started.
  • The newly created task is in relationship "Parent/ Child" with the task from which it is created. 


  •  The task is displayed in the project schedule as an entry of type "Task" (sub-task of the entry of type "Task"). 

Task as a Sub-task of another Workflow Task

The user is able to create sub-tasks of workflow tasks. This is needed because sometimes the execution of one workflow task requires split into sub-tasks and participation of other people.

Rules:

  1. I can set-up the type of tasks that I create as sub-tasks of workflow task
  2. I can set-up if the sub-tasks of the workflow task to block/ not block the user to complete the workflow task
  3. If the the sub-tasks are set up to block the workflow task - then the workflow task could not be completed unless the sub-tasks are in final state (completed or stopped)
  4. If the sub-tasks do not block the completion of the workflow task - then the workflow task could be completed at any time, no matter of the state of the sub-tasks
  5. All sub-tasks of the workflow tasks are visible in the workflow landing page.
  6. A sub-task of a workflow task could have children sub-tasks as well
  • For example we may have a workflow that is started in a case of a project. That workflow may consists of three tasks.

The three tasks of the workflow here are: Review and approve document by first viewer, Review and approve document by second viewer, and Publish document.

  • When the user is on the first task, they may create a sub-task to the workflow task (in this case it is Review and approve document by first reviewer). Since there is a created rule that determines that the workflow task can only be completed when all its sub-tasks are completed or stopped, the Review and approve document by first viewer task  cannot still be completed.

The Approve and Reject buttons are inactive because the Standalone Task 1 is completed or stopped.

 

  • The user completes Standalone Task 1 sub-task.

The workflow task can now be completed by selecting either Approve or Reject button - the two buttons are active, because the Standalone Task 1 sub-task is already completed.

 

  • The user selects the Approve button on the workflow task. Then the Review and approve document by second viewer workflow task is created.
  • The user creates a sub-task Standalone Task 2 to the latter workflow task. Since there is a created rule that determines that the workflow task can be completed irrespective of the status of any of its sub-tasks, the Review and approve document by second viewer task  can be completed even when Standalone Task 2 is in Approved or in In Progress status.

The Standalone Task 2 sub-task is in active status (Approved) and the workflow task can be completed by selecting either Approve or Reject button - the two buttons are active.

  • The third workflow task, Publish document, can have any of the two rules about sub-tasks that are mentioned above.

In summary, the ability to set rules on how a task can be completed based on the status of its sub-tasks gives more flexibility when a task is assigned to a person or to a group of people.
