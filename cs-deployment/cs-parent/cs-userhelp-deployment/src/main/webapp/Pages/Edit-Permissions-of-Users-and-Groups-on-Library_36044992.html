Sirma Platform : Edit Permissions of Users and Groups on Library

Administrators in the system (set with manager permissions) are able to edit/ add/ remove permissions to some users and groups for every domain library in the system.

The permissions that are set on library level are inherited from all entities in library, which are with flag “Inherited permissions” or “Inherited and Special Permissions”.

A "library" is actually the place where we see with all entities from one and the same class. Giving permissions on a library means to give permissions on the respective class.

The "class" as a concept is accessible only from the administrators, so that the normal users do not get confused. See Manage Class Landing Page for more details on the classes and their landing pages.

When an user doesn't have permissions (set with "No permissions") on a library, he/she is not able to see the respective library in the Library menu. The ontology class is not visible in the search screen and the user cannot search by objects from this type.

Rules:

  1. Permissions on the library are inherited from each object in the library, which is with flag “Inherited permissions” or “Inherited and Special Permissions”.
  2. Permissions on the library are inherited from the children documents of the objects in the library (the uploaded images, scanned documents, etc.), which are with inherited permissions.
  3. Objects with flag “Special permissions” do not inherit the default library permissions.

    • Add/ remove user/ group and select role
  • Any object in the library inherits the permissions set on the library

Edit permissions on library

  • The administrator opens a library selecting it from the Objects library menu (1-2).

  • The administrator selects Library Details from the Actions menu (1-2).

  • The system opens the library landing page. The administrator opens the Permissions (1) tab and clicks Edit Permissions button (2).

Add/ remove user/ group and select role

  • The administrator adds user/ group selecting it from a drop-down list (1-2-3-4).

  • The selected user/group appears in the corresponding field (1). The administrator selects role for the added user/ group selecting it from list (2-3).

    • The permissions from which the administrator could select on a library, and give to some users and groups are:

      • No permissions – could not see the object library at all and the objects in the library, even if makes searches in the system
      • Consumer – could only view the objects in the library, cannot edit them, or upload content to them, or edit uploaded content, but could start workflows and tasks, could attach to cases and projects
      • Contributor – could create new objects and edit/ delete only the objects he created
      • Collaborator – could create new objects and edit objects created by others, but could delete  objects created by himself
      • Manager – has full permissions on the library – could edit/ delete objects created by himself and the other users, could manage the permissions on the library

  • The selected role for the added user/group appears in the corresponding field (1).  The administrator could change the roles of users/ groups choosing new role from the list.
  • The administrator could remove user/group from the permissions section by clicking the corresponding button (2).
  • The Administrator clicks Save Permissions (3).

  • The changes of permissions on the library are saved and appear in the corresponding section in the library landing page (1-2).

Any object in the library inherits the permissions set on the library

  • The administrator opens the landing page of an object created in the library (1-2).
  • The permissions on the object in the library inherit the permissions set on the library (3-4-5).

(Not Ready) Restore the permissions of all objects in the library to current library permissions

As an Administrator in the system I want to be able easily to restore the permissions of all entities in a library to the current library permission and thus to remove all existing special permission on single objects, in case there is a need to change to security policy in the organization.
