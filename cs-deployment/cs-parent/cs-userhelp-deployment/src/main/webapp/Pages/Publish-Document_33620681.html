Publish Document Manually

The user is able to publish manually a document (could be any type of document – iDoc, image, pdf, etc) in order to have a locked revision of the document at that very moment.

The published document is an exact copy of the respective version of the original document. The permissions to the published revision may be set different than the permissions on the original document.

Depending on the definitions set for the document, some document types may to be published when in Draft state and other document types to be published when in Approved or Rejected state.

Rules:

  1. One document could be published as many times as the user wants.
  2. Only one approved revision of a document could exist at a certain point of time (and many obsolete and rejected revisions).
  3. Permissions set on each of the published revision could be different than the permissions of the original document.
  4. The published revision of a document is locked and no one could edit it.
  5. If a document is an iDoc, then the widgets of its published revisions never get updated with new content.
  6. When approved or rejected document is published, then its status is transferred as a status of the revision and the original document changes its status back to draft, which makes it editable again.
  7. When a document is published the version of the original document is increased to the next major version.

Publish Draft document for first time

The document in status "Draft" and there is no existing revision of the current document.

  • The user select "Publish document" from the list with actions on the document.

  • The system provides the user with an option to select the location where the revision to be attached (1-2-3-4-5-6).

  • The user confirms the operation.

  • The version and the revision of the original document are changed (1). 
    • The status of the document remains "Draft" (2). 
    • Relationship of type "has revision/ is revision of" is created between the document and the document revision (3-4).

  • A new object of type "Document revision", cloning of the current document, is created in status "Approved" with Revision 1 in the selected location (1-2-3-4-5). The published revision is locked for edit (6).

  • The action is recorded in the Audit Log.

Publish Draft document for a second time

  1. The current document is in status "Draft", version 2.0 Revision 1
  2. There is one existing revision of the current document - Revision 1 in status Approved.
  • The user selects "Publish document" from the list with actions on the document, selects the location of the next revision and confirms the operation.
  • The version of the original document changes to 3.0 (1). Relationship of type "has revision/ is revision of" is created between the document and the document revision 2 (2-3). The status on the original document is "Draft"(4).


  • A new object of type "Document revision", cloning of the current document (version 2.4), is created in status "Approved" with Revision 2.

  • The status of Revision 1 changes to "Obsolete".

Publish Approved or Rejected documents

Approve Document (manually)

  • The user is able to approve document selecting the corresponding action from the actions drop-down menu on the landing page of document which is in status "Draft" (1-2).

  • The version of the document does not change (1). The document changes its status to "Approved" (2). Possible actions on the approved document are only "Publish Document" and "Download Document" (3).

Reject Document (manually)

  • The user is able to reject document selecting the corresponding action from the actions drop-down menu on the landing page of document which is in status "Draft" (1-2).
  • The version of the document does not change. The document changes its status to "Rejected". Possible actions on the rejected document are only "Publish Document" and "Download Document".

Example 1: Publish a Rejected Document

  • The user opens the landing page of document in status "Rejected", Version 1.0 and selects the action "Publish Document", selects location for the revision and confirms the operation.

  • The version of the original document changes to 2.0 and the status changes to "Draft".  Relationship of type "has revision/ is revision of" is created between the document and the document revision 1.

  • A new object of type "Document revision", cloning of the current document (version 2.4), is created in status "Rejected" with Revision 1.

Example 2: Publish an Approved Document with more then one revisions

  • The user opens the landing page of document in status "Approved", Version 3.1,Revision 2. (1-2).
    • There is Revision 1 of the document in status "Approved" (3).
    • There is Revision 2 of the document in status "Rejected" (4).
  • The user selects the action "Publish Document" (5-6), selects location for the next revision and confirms the operation.

 

  • A new object of type "Document revision", cloning of the current document, is created in status "Approved" with Revision 3.

  • The version of the original document changes to 4.0 (1) and the status to "Draft" (2).
  • Relationship of type "has revision/ is revision of" is created between the document and the document Revision 3 (4-5).

  • The status of Revision 1 changes to "Obsolete".

  • The status of Revision 2 remains "Rejected".

Publish Document and its Sub-documents

When the user publishes a document, and the document has sub-documents, then the document is published together with its child documents. The revisions of the main document and the child documents update synchronously.

The child documents cannot be published independently from the parent document. The action "Publish document is active only for the parent document.

Example:

  • A sub-document (autocreated doc.png) version 1.0 (1-2) is uploaded in a document (ECN25858.DOC). The automatically created relation (Part of and Has Parent) are visible (3) in the Relations tab of the child sub document .

  • The user opens the parent document (ECN25858.DOC) version 1.0 (1-2)
    • The automatically created relations (Has child and Parent of) are visible in the relations tab of the parent document (3). 
  • The user selects the action "Publish Document" (4-5).

  • The user confirms the operation.

  • The parent document is published together with its sub-document.
    • The version of the main document changes to version 2.0 (1). The version is updated in the "Parent of" relationship (2).
    • Relations "Last published revision" (3) and "Last revision" (4) are automatically created.

    • The version of sub-document changes to version 2.0 (1).  The version is updated in the "Part of"(2) and "Has Parent" (3) relationships .
    • Relations "Last published revision" (3) and "Last revision" (4) are automatically created.

 

Publish Document Automatically from workflow

In many cases a document that is approved/ rejected as part of a workflow for review and approval should also be published.

The user is able to publish a document automatically with its approval or rejection from a workflow.

Rules:

  1. There is one action "Approve and Publish" (Approve/ Complete) or "Reject and Publish" (Reject) in the workflow.
  2. There is no action "Publish Document" on the document.

Approve and publish document in workflow

Preconditions:

  1. A workflow for review and approval is running on a document (1).
  2. The document is in status "In process", version 1. There is no existing revision of the current document (2).
  3. There is a task assigned to the user to approve the document and Complete the workflow.
  • The user selects the button "Complete" (3) in the opened task.

  • The system opens a screen with an option to select the location where the revision to be attached. The user selects the button "Attach Location".
    • If no location is selected and button "OK" is clicked the revision would be created as a child to the original document.

  • The user selects the location where the revision to be published (1-2-3-4-5-6).

  • The selected location where the revision will be attached (1) is displayed in the confirm panel. The user clicks "OK" (2).

  • The version of the original document changes to 2.0 (1). The state of the original document is "Draft" (2). Relationship of type "has revision/ is revision of" is created between the document and the document revision (3-4).

  • A new object of type "Document revision", cloning of the original document is created in status "Approved" with Revision 1(1-2-3-4). The revision is locked for edit (5).

Reject and publish document in workflow

Notes:

  •  When a workflow is run on a document that has no existing revisions and a workflow task for approval or rejection is Rejected then a new object of type "Document revision", cloning of the current document, is created in status "Rejected" with Revision 1.
  • When a new workflow is run on a document that has one existing revision of the document (Revision 1, Approved) and a workflow task for approval or rejection is Rejected then:
    • The approved Revision 1 remains unchanged (it is still the last approved revision).
    • A new object of type "Document revision", cloning of the current document, is created in status "Rejected" with Revision 2.
