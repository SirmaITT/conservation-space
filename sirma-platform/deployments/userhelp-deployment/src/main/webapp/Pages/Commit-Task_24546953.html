Sirma Platform : Commit Task

The PM or the team leads are able to create task instances based on the task details from the project timeline and assign tasks to users.
The project schedule (timeline) by itself is just a plan, a work break down structure of the tasks per project. The assignees don't know about their tasks and don't have assignments for them.
The "Commit task" functionality aims each user to receive a direct task, assigned to him/her and the PM to be able to track progress on tasks. 

For each task in the project schedule in its details is kept information about how and when a task will be started - the field name is "Trigger to start". The following options are available:

  1. Manual start - this is the default option with which the task is created. Manual start means that the user must explicitly select a task and start it.
  2. Automatically, based on the task links - this option means that the task will be started based on its predecessor task. If the link with the predecessor is Start to Start then the task will be started when its predecessor starts. If the link with the predecessor is End to Start, then the task will be started when its predecessor finishes.
  3. Automatically at start date - this option means that the task will be started at the date it is planned for. It is appropriate for tasks which do not depend on other tasks, or for tasks that do not have predecessors but only successors.

For a task in status "Submitted" the user could change "Trigger to start" at any time (see Edit task details).
Each task in status "Submitted" could be manually started at any time (if not started), no matter what is the value in the "Trigger start" field.

Preconditions

  1. Commit tasks could not be performed on sub-tasks of "Workflow". 
  2. The action "Commit task" could be performed on Tasks in status "Submitted".
  • The user opens the project schedule and selects single or multiple tasks and then button "Commit task(s)" from the opened menu.

  • The system opens a confirmation screen (1) and the user selects the button "Yes" (2).

  • The system creates a task instance with task details from the task in the project schedule: task name, task type, planned start date, planned end date, duration, assignee, etc.
    • Unique task ID (1) is generated to the newly created task. The Task ID is assigned also to the task in the project schedule.
    • The task ID of the created task instance is displayed in the project schedule in a separate column. The ID is selectable link and the task instance could be opened when the user clicks on the ID.
    •  The newly created task is in status "Approved" (2).

  • The committed (started) task  appears in the "Project dashboard" in the corresponding dashlet  " Project Tasks" (for committed task of type "Task" and started tasks in task of type "Workflow") or  in "Cases" dashlet (for committed task of type "Case").
